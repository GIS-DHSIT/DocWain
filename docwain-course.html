What Does DocWain Actually Do?

Imagine uploading a 200-page report and asking it a question in plain English. Here's what happens under the hood.

The Elevator Pitch

DocWain is a web application that reads your documents, understands what's inside them, and answers your questions using only the information from your documents — not the internet, not its training data.

💡
Why this matters: Most AI chatbots make things up. DocWain is grounded — every answer must be backed by evidence from your actual documents. If it can't find the answer in your files, it says so.

DocWain uses FastAPI to listen for incoming requests. When the server boots up, it pre-loads everything it needs:

1

A 335-million-parameter embedding model that converts text into numbers for searching.

2

Connect to Databases

Qdrant (vector search), MongoDB (metadata), and Redis (cache).

3

Initialize LLM Clients

Connects to the AI language models (local Ollama, cloud Azure, or Gemini) that generate answers.

🔑
CS Insight — Lazy Loading: DocWain doesn't load everything at once. Heavy components like the AI models are loaded in parallel threads during startup, so the server starts faster. It's like a restaurant opening: the kitchen starts prepping while the host sets tables — they don't wait for one to finish before starting the other.

chunk_text_stream_with_metadata() — Streaming

CODE 
# execution/common.py:116-150
def chunk_text_stream_with_metadata(
    text, metadata=None, chunk_size=256):
    pos = 0
    while pos < len(text):
        end = min(pos + chunk_size, len(text))
        chunk = text[pos:end]
        last_nl = chunk.rfind('\n')
        last_sent = chunk.rfind('. ')
        best_break = max(last_nl, last_sent)
        if best_break > chunk_size * 0.4:
            end = pos + best_break + 1
        yield text[pos:end]
        pos = end
    yield '\n\n[META]' + json.dumps(metadata)
PLAIN ENGLISH

Break a long answer into ~256-character chunks for real-time streaming (the "typing" effect in the UI).

Smart splitting: don't chop mid-sentence. Look for the best break point: newline > sentence end > table row.

Only use the break point if it's at least 40% into the chunk (prevents tiny chunks).

Yield each chunk one at a time — the HTTP connection stays open and the user sees words appear progressively.

After all text, send a final [META] chunk with grounding status and sources so the frontend can show citation badges.

Step 5: Deduplication — No Repeats

Sometimes documents repeat themselves — headers, footers, boilerplate. DocWain uses a clever fingerprinting technique called Simhash to catch duplicates:

CODE 
def _simhash64(text: str) -> int:
    tokens = _normalize_text(text).split()
    vector = [0] * 64
    for token in tokens:
        h = hashlib.md5(token).digest()
        for i in range(64):
            bit = 1 if (h >> i) & 1 else -1
            vector[i] += bit
PLAIN ENGLISH

Create a 64-bit fingerprint for any piece of text.

Split the text into individual words.

Start with a blank scorecard: 64 slots, all zeros.

For each word, generate a unique hash (like a word's DNA).

Each bit of the hash votes: +1 or -1 on each slot.

After all words vote, the positive slots become 1s, the rest 0s — that's the fingerprint.

Two chunks with ≥92% similar fingerprints are treated as duplicates.

Dense Vector Search

Your question gets converted into the same kind of vector as the document chunks. Then Qdrant finds the closest matches — like a GPS finding the nearest restaurants to your location, except in 1,024-dimensional space.

CODE 
def retrieve(self, query, subscription_id,
             profile_ids, top_k=30):
    query_vector = self.embedder.encode([query])[0]
    
    must = [
      FieldCondition(key="subscription_id",
        match=MatchValue(value=subscription_id)),
      FieldCondition(key="profile_id",
        match=MatchValue(value=profile_id)),
    ]
PLAIN ENGLISH

Search for the best 30 matching chunks for a question.

First, convert the question into a vector (list of numbers) — the same format used for stored chunks.

Then set up security filters: only search within this customer's subscription AND this specific profile.

This ensures Customer A can never accidentally see Customer B's documents — even if their vectors are similar.

💡
Why "top 30"? DocWain intentionally retrieves more chunks than it needs. It's like casting a wide fishing net — grab 30 candidates, then let the reranker pick the best 3-5. This two-stage approach is more accurate than trying to find the perfect results in one shot.

The Keyword Safety Net

Sometimes vector search misses obvious matches — like when your question uses the exact same rare word as the document. DocWain has a fallback:

1

Dense search runs first

The vector-based search finds chunks by meaning. Results scoring ≥ 0.5 are considered "high quality."

2

Check: enough good results?

If fewer than 3 high-quality results were found, the system suspects it might be missing something.

3

Keyword fallback kicks in

Scrolls through chunks looking for exact word matches, scored by F1 overlap. These keyword results are scored at 50% (0.5×) to rank below the vector results.

Reranking — The Expert Second Opinion

Vector search is fast but approximate — like a quick Google search. The cross-encoder reranker is slower but much more precise — like having an expert read each result carefully.

Two-Stage Search
30 candidates from vector search
↓ Cross-encoder compares each one to the original question
Re-scored and re-ordered by relevance
↓ Top 3-5 chunks selected
Best evidence sent to the LLM
🔑
CS Insight — The Funnel Pattern: This "wide then narrow" pattern appears everywhere in engineering. Airport security: thousands enter, scanners filter, then manual checks for flagged passengers. Same idea — cheap first pass, expensive second pass only on survivors.

_keyword_fallback() — The Safety Net Algorithm

CODE 
# retriever.py:189-235
def _keyword_fallback(self,
    collection_name, query, qfilter,
    top_k, existing_ids):
    records = self.qdrant_client.scroll(
        collection_name, scroll_filter=qfilter,
        limit=top_k, with_payload=True)[0]
    query_tokens = set(query.lower().split())
    for record in records:
        text_tokens = set(text.lower().split())
        overlap = query_tokens & text_tokens
        precision = len(overlap)/len(query_tokens)
        recall = len(overlap)/len(text_tokens)
        f1 = 2*precision*recall/(precision+recall)
        chunk.score = f1 * 0.5  # Half-weight
PLAIN ENGLISH

Scroll through chunks matching the security filter (no vector math, just raw iteration).

Split the query into individual words. Split each chunk's text into words.

Find which words overlap between query and chunk text (set intersection).

Calculate F1 score: balances precision (did I match the right words?) and recall (did I find all query words?).

Score keyword results at 50% (× 0.5) so they always rank below vector results. This ensures keyword matches supplement, never override, the primary search.

Skip chunks already found by dense search (existing_ids) to avoid duplicates.

_point_to_chunk() — Qdrant → EvidenceChunk

# retriever.py:237-263
@staticmethod
def _point_to_chunk(point, profile_id):
    payload = point.payload or {}
    chunk_meta = payload.get("chunk") or {}
    section_meta = payload.get("section") or {}
    provenance = payload.get("provenance") or {}
    text = payload.get("canonical_text")
        or payload.get("embedding_text")
    return EvidenceChunk(
        text=text,
        source_name=provenance.get("source_file"),
        section=section_meta.get("title"),
        score=getattr(point, "score", 0.0))
PLAIN ENGLISH

Convert a raw Qdrant database record into our EvidenceChunk format.

Qdrant stores metadata as a nested payload with three sections: chunk (ID, type), section (title, path), and provenance (page numbers, source file).

Prefer canonical_text (clean, de-duplicated text) over embedding_text (the text used for vector encoding). They're usually the same, but canonical_text has post-processing applied.

The score comes from Qdrant's cosine similarity calculation — how close the query vector was to this chunk's vector.

def _extract_person_from_query(query):
    import spacy
    if _ner_nlp_cache is None:
        _ner_nlp_cache = spacy.load(
            "en_core_web_sm")
    doc = _ner_nlp_cache(query)
    for ent in doc.ents:
        if ent.label_ == "PERSON":
            return ent.text.strip()
    for ent in doc.ents:
        if ent.label_ == "ORG":
            return ent.text.strip()

Find the name of a person (or organization) mentioned in the query.

Fallback: ORG entities ("Google", "Microsoft") for vendor/company queries.

classify_conversational_intent() — The Gatekeeper

CODE 
# conversational_nlp.py:53-122
def classify_conversational_intent(
    text, turn_count=0):
    # Possessive entity guard
    _possessive_re = re.compile(
        r"[A-Z][a-z]{2,}'s?\s+\w+")
    if _possessive_re.search(text):
        return None  # Document query!

    routing, intent, score = \
        classify_query_routing(text)
    _THRESHOLD = 0.65
    if score < _THRESHOLD
        and intent not in _CLEARLY_CONV:
        return None  # Route to RAG
    if intent == GREETING and turn_count > 0:
        return (GREETING_RETURN, score)
PLAIN ENGLISH

The most critical routing decision: is this a conversational message or a document query?

Possessive entity guard: "Philip's skills", "Sarah's education" — even though these are short, the capital letter + apostrophe-s pattern signals a person's data. Always route to document search.

Use the NLU engine's contrastive embedding classifier to score the query against conversational prototypes vs document-query prototypes.

Confidence gate at 0.65: borderline queries (score < 0.65) default to document search. "Can you show me..." sounds conversational but is usually a document request. Better to search and find nothing than to skip search incorrectly.

Exception: clearly conversational intents (greeting, farewell, thanks, praise) bypass the confidence gate — they're safe to handle without documents.

Returning user detection: If this is a GREETING but turn_count > 0 (they've already chatted), promote to GREETING_RETURN for a different response ("Welcome back!" instead of "Hi, I'm DocWain!").

def generate_with_metadata(self, prompt,
    *, options=None, max_retries=3,
    thinking=False):
    generation_options = {
        "temperature": 0.2,
        "top_p": 0.85,
        "num_ctx": 8192,
        "num_predict": 4096,
    }
    extra_kwargs = {"think": thinking}
    for attempt in range(1, max_retries+1):
        response = self._client.generate(
            model=self.model_name,
            prompt=prompt,
            options=generation_options)
        text = response.response.strip()
        if thinking and "<think>" in text:
            reasoning, text = _split_thinking(text)
PLAIN ENGLISH

Send a prompt to the Ollama LLM and get a response with metadata.

Generation parameters: temperature 0.2 (low = factual, not creative), top_p 0.85, context window 8192 tokens, max output 4096 tokens.

Retry loop: try up to 3 times. If the model is loading into GPU (detected by "loading model" in error), wait 15 seconds and try once more.

Thinking mode: When enabled, the model wraps its reasoning in <think>...</think> tags. This function extracts the reasoning separately so the UI can show it as a collapsible "thought process" section.

Metrics: Records latency, retry count, and model usage to the metrics store for monitoring.

_split_thinking() — Extracting AI Reasoning

CODE 
# llm/clients.py:52-61
_THINK_RE = re.compile(
    r"<think>(.*?)</think>", re.DOTALL)

def _split_thinking(text):
    match = _THINK_RE.search(text)
    if not match:
        return "", text
    reasoning = match.group(1).strip()
    cleaned = _THINK_RE.sub("", text).strip()
    return reasoning, cleaned
PLAIN ENGLISH

Some AI models (like Qwen3) wrap their reasoning in <think> tags before giving the final answer.

This regex finds the <think>...</think> block, extracts the reasoning inside it, and removes it from the visible response.

Returns a tuple: (reasoning trace, clean answer). The reasoning can be shown in a debug panel; the clean answer goes to the user.

Trust But Verify

How DocWain ensures every answer is accurate, every document is safe, and no claims are made without evidence.

The Evidence Gate

After the AI generates an answer, it doesn't go straight to the user. First, it passes through the Evidence Gate — a fact-checker that validates every claim.

1

Build the Evidence Ledger

Map every [SOURCE-1], [SOURCE-3] citation in the answer back to actual chunk snippets from the documents.

2

Sentence-Level Validation

Each sentence is checked: Does it cite a source? Does the source actually support the claim? Is there ≥35% token overlap between the claim and evidence?

3

Remove Unsupported Claims

Any sentence that fails validation is silently removed. The user only sees claims backed by real document evidence.

build_evidence_ledger() — Mapping Citations to Sources

CODE 
# response_policy.py:313-355
def build_evidence_ledger(chunks, sources):
    entries = []
    for source in sources:
        source_id = int(source.get("source_id"))
        matched_chunk = _match_source_to_chunk(
            source, chunks)
        snippet_text = _chunk_text(matched_chunk)
        tokens = _content_tokens(
            _tokenize(snippet_text))
        numbers = _extract_numbers(tokens)
        entries.append(EvidenceSnippet(
            source_id=source_id,
            exact_snippet=snippet_text,
            tokens=tuple(tokens),
            numbers=tuple(numbers)))
PLAIN ENGLISH

Build a cross-reference table that maps each [SOURCE-N] citation in the AI's answer to the actual document chunk it references.

For each source citation: find the best-matching chunk using _match_source_to_chunk() (matches by source filename + excerpt text overlap).

Tokenize the matched chunk's text: split into words, remove stopwords ("the", "a", "is"), extract numbers separately.

Store as an EvidenceSnippet: the source ID, exact text, content tokens, and numbers.

This ledger is then used by the Evidence Gate to validate that each claim in the answer is actually supported by the cited source.

_match_source_to_chunk() — Finding the Right Evidence

CODE 
# response_policy.py:286-311
def _match_source_to_chunk(source, chunks):
    source_name = source.get("source_name")
    excerpt = source.get("excerpt")
    best_chunk = None
    best_score = 0.0
    for chunk in chunks:
        score = 0.0
        if source_name == chunk_name:
            score += 2.5  # Filename match
        if excerpt in chunk_text:
            score += 3.0  # Exact excerpt match
        else:
            overlap = excerpt_tokens & chunk_tokens
            score += len(overlap)/len(excerpt)
        if score > best_score:
            best_chunk = chunk
PLAIN ENGLISH

When the AI says "[SOURCE-2]", find which actual document chunk it's referring to.

Scoring system: each chunk gets a match score based on two signals:

+2.5 points if the source filename matches the chunk's source filename (strong signal).

+3.0 points if the source excerpt appears verbatim inside the chunk text (exact match = highest confidence).

If no exact match, fall back to token overlap ratio: how many words in the excerpt also appear in the chunk?

Return the chunk with the highest combined score. If no chunk scores above 0, return None (citation is orphaned).
