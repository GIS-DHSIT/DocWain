What is DocWain?

Before we look at any code, let's understand what this platform does and why it exists.

The Elevator Pitch

DocWain is an AI-powered document intelligence and RAG platform. It reads your documents, understands what is inside, and answers questions using ONLY your document evidence.

⚠
Why this matters: Most AI chatbots make things up. DocWain is grounded — every answer must be backed by evidence from your actual documents. If it cannot find the answer in your files, it says so honestly.

The Three-Step Promise

Upload Documents
→
Ask Questions
→
Get Grounded Answers

Key Capabilities

📄

Multi-Format

Supports PDF, DOCX, PPTX, Excel, CSV, JSON, and plain text. Extracts text from images via OCR.

🎯

Grounded Answers

Every answer is backed by evidence from your documents. No hallucination — if it cannot find proof, it tells you.

🏢

Multi-Tenant

Serves multiple organizations from a single deployment. Each tenant's data is fully isolated via subscription_id + profile_id.

🤖

Microsoft Teams

Integrates directly into Teams chat. Upload documents and ask questions without leaving the Teams interface.

📈

Self-Improving

Fine-tuning loops (SFT, DPO, behavioral scoring) continuously train the model on real usage data to improve accuracy.

🔒

Document Screening

Automatic security screening: PII detection, integrity checks, compliance validation, AI authorship detection.

Multi-Tenancy Model

All data is scoped by two keys that form a hierarchical isolation boundary:

1

subscription_id

Identifies the organization or tenant. Maps to a Qdrant collection. All documents for one organization live in one collection.

2

profile_id

Identifies a specific document group within the organization. Enforced via payload filters on every query. One subscription can have many profiles (e.g., one per department or project).

The Two Pipelines

Everything DocWain does boils down to two pipelines: ingesting documents and answering questions.

Pipeline 1: Document Ingestion

When a user uploads a document, it goes through this pipeline before it becomes searchable.

1

Upload

Document arrives via API or Teams. Stored in Azure Blob Storage for persistence.

2

Extract

Text, tables, and images extracted from the document. OCR via pytesseract or vision LLM for scanned pages.

3

Chunk

Text is split into overlapping chunks (250-450 tokens, 60 token overlap). Section-aware chunking preserves document structure.

4

Screen

Security screening runs: PII detection, integrity hash, compliance checks, AI authorship detection. Results stored with the document.

5

Embed

Each chunk is converted to a 1024-dimensional vector using BAAI/bge-large-en-v1.5.

6

Dedup

Duplicate chunks removed via Simhash64 with threshold 0.92. Prevents the same paragraph from appearing multiple times.

7

Store

Vectors stored in Qdrant with full metadata payload (subscription_id, profile_id, document_id, section info, provenance).

Pipeline 2: Question Answering

When a user asks a question, it goes through this pipeline to produce a grounded answer.

1

Question Arrives

User's question hits /api/ask with subscription_id, profile_id, and the query text.

2

Intent Analysis (UNDERSTAND)

The IntentAnalyzer classifies the query: What task type? What output format? What entities are mentioned? Is it conversational or a document query?

3

Retrieve (RETRIEVE)

The UnifiedRetriever searches Qdrant for the most relevant document chunks using vector similarity. Falls back to keyword search when needed.

4

Rerank

A cross-encoder reranker re-scores the top results for higher precision. The best evidence bubbles to the top.

5

Generate (REASON)

The Reasoner constructs a prompt with the evidence and calls the LLM. For complex queries, uses Qwen3 thinking mode for step-by-step reasoning.

6

Evidence Gate

The evidence gate verifies grounding. Every claim must be traceable to retrieved evidence. If coverage is below threshold, the response is flagged.

7

Compose & Respond (COMPOSE)

The answer is sanitized (remove AI disclaimers, fix OCR artifacts), formatted per the detected output format, and returned with sources and grounding metadata.

Side by Side

Both pipelines operate independently. Ingestion runs when documents are uploaded. Question answering runs when users ask questions. They share the same storage layer.

Ingestion Pipeline

Runs on document upload. Writes to Qdrant, MongoDB, Azure Blob. Makes documents searchable.

QA Pipeline

Runs on user questions. Reads from Qdrant, generates answers via LLM, validates grounding.

vector_store.py — Qdrant Helpers

CODE 
def build_collection_name(subscription_id: str):
    safe_sub = str(subscription_id).strip()
        .replace(" ", "_")
    return safe_sub

def build_qdrant_filter(sub_id, profile_id, doc_ids):
    must = [
        FieldCondition(key="subscription_id",
            match=MatchValue(value=sub_id)),
        FieldCondition(key="profile_id",
            match=MatchValue(value=profile_id))]
    if doc_ids:
        must.append(FieldCondition(key="document_id",
            match=MatchAny(any=doc_ids)))
    return Filter(must=must)
PLAIN ENGLISH

build_collection_name — converts a subscription_id into a safe Qdrant collection name (spaces to underscores).

build_qdrant_filter — builds a Qdrant filter that enforces tenant isolation. Every search MUST include subscription_id and profile_id. Optionally narrows to specific document_ids.

This is the core of multi-tenancy: these filters ensure a query in Profile A never sees vectors from Profile B.

CODE 
def ensure_collection(client, name, vector_size):
    with _get_collection_lock(name):
        if client.collection_exists(name):
            # Validate schema matches
            info = client.get_collection(name)
            if info.config.params.vectors.size != vector_size:
                raise QdrantCollectionSchemaMismatch(...)
            return
        client.create_collection(name,
            vectors_config={"content_vector":
                VectorParams(size=vector_size,
                    distance=Distance.COSINE)})
PLAIN ENGLISH

ensure_collection — creates a Qdrant collection if it does not exist, or validates it if it does.

Uses a per-collection lock to prevent race conditions when multiple requests try to create the same collection simultaneously.

If the collection exists but has a different vector size, raises a schema mismatch error (critical safety check).

Vectors use cosine distance for similarity scoring.

core_agent.py — CoreAgent Class

CODE 
class CoreAgent:
    MAX_SUBAGENTS = 5
    SUBAGENT_TIMEOUT = 30.0

    def __init__(self, llm_gateway, qdrant_client,
                 embedder, mongodb, kg_query_service,
                 cross_encoder):
        self._intent_analyzer = IntentAnalyzer(llm_gateway)
        self._retriever = UnifiedRetriever(qdrant_client, embedder)
        self._reasoner = Reasoner(llm_gateway)
        self._domain_dispatcher = DomainDispatcher(llm_gateway)
PLAIN ENGLISH

CoreAgent is the brain of DocWain. It ties the entire pipeline together.

On creation, it wires up four sub-components: IntentAnalyzer (understands what the user wants), UnifiedRetriever (searches documents), Reasoner (generates answers via LLM), and DomainDispatcher (handles specialized domain tasks).

For complex queries, it can spawn up to 5 parallel sub-agents with a 30-second timeout each.

CODE 
def handle(self, query, subscription_id, profile_id,
           user_id, session_id, conversation_history,
           *, agent_name=None, document_id=None,
           debug=False):
    # --- UNDERSTAND ---
    doc_intelligence = self._load_doc_intelligence(sub_id)
    kg_hints = self._kg_probe(query, sub_id, profile_id)
    understanding = self._intent_analyzer.analyze(
        query, doc_intelligence, conversation_history)

    # --- RETRIEVE ---
    evidence = self._retriever.retrieve(
        query=understanding.resolved_query,
        subscription_id=sub_id,
        profile_ids=[profile_id], top_k=top_k)

    # --- REASON ---
    result = self._reasoner.reason(
        query, evidence.chunks, understanding)

    # --- COMPOSE ---
    return compose_response(result, evidence, understanding)
PLAIN ENGLISH

handle() is the main entry point — every question flows through here.

UNDERSTAND: Load document intelligence (summaries, topics). Probe the knowledge graph for entity hints. Run intent analysis to classify the query (task type, complexity, output format, entities).

RETRIEVE: Search Qdrant for the most relevant document chunks using the resolved query. The number of results (top_k) varies by task type — lookups need 4 chunks, overviews need 10.

REASON: Feed the query and evidence to the LLM via the Reasoner. For complex queries, uses Qwen3 thinking mode for step-by-step reasoning.

COMPOSE: Format the final response with sources, grounding metadata, and the answer text sanitized of AI artifacts.

CODE 
class IntentAnalyzer:
    def analyze(self, query, doc_intel, history):
        # Fast-path: check for conversational queries
        if _GREETING_RE.match(query):
            return QueryUnderstanding(
                task_type="conversational", ...)
        # Full analysis via LLM prompt
        prompt = build_understand_prompt(
            query, doc_intel, history)
        raw = self._llm.generate(prompt)
        return self._parse_understanding(raw)
PLAIN ENGLISH

IntentAnalyzer.analyze() replaces six competing intent classifiers with one unified system.

Fast-path: Greetings ("hi", "hello"), farewells, and meta questions ("who are you") are detected instantly via regex — no LLM call needed.

Full analysis: For document queries, builds a structured prompt with the query, document summaries, and conversation history, then asks the LLM to classify task type, complexity, output format, and relevant entities in one call.

domain_dispatch.py — DomainDispatcher

CODE 
class DomainDispatcher:
    def try_handle(self, query, sub_id, profile_id,
                     evidence, doc_context, *,
                     agent_name=None):
        detection = self._detect(query, agent_name)
        if detection is None:
            return None  # Not a domain task
        agent = get_domain_agent(detection["domain"])
        result = agent.execute(
            detection["task_type"], context)
        return {"response": result.output,
                "grounded": True,
                "metadata": {"agent": domain}}
PLAIN ENGLISH

DomainDispatcher sits between UNDERSTAND and RETRIEVE in the pipeline.

It checks if the query matches a specialized domain (e.g., resume screening, medical analysis).

If a domain agent exists and can handle it, it bypasses the standard RAG pipeline and delegates directly to the specialized agent.

If not, returns None — the query falls through to the standard RETRIEVE/REASON pipeline.

deterministic_router.py — Query Classification

CODE 
def _detect_task_type(query: str) -> str:
    from src.nlp.nlu_engine import classify_intent
    intent = classify_intent(query)
    _INTENT_TO_TASK = {
        "comparison": "compare",
        "ranking": "rank",
        "summary": "summarize",
        "factual": "qa",
        "timeline": "timeline",
    }
    task = _INTENT_TO_TASK.get(intent, "qa")
    # Check for generation intent
    sem = parse_query(query)
    if any(v in ("generate", "create", "write")
           for v in sem.action_verbs):
        return "generate"
    return task
PLAIN ENGLISH

_detect_task_type classifies what the user wants to do using spaCy NLU.

Maps NLU intents (comparison, ranking, summary, factual) to task types (compare, rank, summarize, qa).

Special overrides: if the query contains action verbs like "generate", "create", "write", it becomes a "generate" task. If it says "list" or "show", it becomes a "list" task.

Also detects temporal queries ("timeline of...", "career progression") and exhaustive detail queries ("all details about...").

CODE 
def _detect_output_format(query, task_type):
    if "json" in query.lower(): return "json"
    if "table" in query.lower(): return "table"
    if task_type == "summarize": return "bullets"
    if task_type == "compare": return "sections"
    if task_type == "extract": return "structured"
    return "free_text"

def _extract_person_from_query(query):
    doc = spacy.load("en_core_web_sm")(query)
    for ent in doc.ents:
        if ent.label_ == "PERSON": return ent.text
    for ent in doc.ents:
        if ent.label_ == "ORG": return ent.text
    return None

def _infer_domain(query, session_state, catalog):
    inferred = infer_domain(query)
    if inferred not in {"unknown", "generic"}:
        return inferred
    if session_state.get("active_domain"):
        return session_state["active_domain"]
    return catalog.get("dominant_domains", "unknown")
PLAIN ENGLISH

_detect_output_format — figures out how the answer should look. Explicit mentions of "json" or "table" win. Otherwise: summaries get bullets, comparisons get sections, extractions get structured output, everything else gets free text.

_extract_person_from_query — uses spaCy NER to find person names in the query ("What certifications does Sarah have?" extracts "Sarah"). Falls back to ORG entities for company queries.

_infer_domain — determines the document domain (resume, legal, medical, invoice). Tries the query first, then falls back to the session's active domain, then to the profile's dominant domain.

conversational_nlp.py — Non-Retrieval Intents

CODE 
def classify_conversational_intent(
    text, turn_count=0):
    # Guard: possessive entity patterns are always
    # document queries ("Philip's skills")
    if _possessive_entity_re.search(text):
        return None  # Not conversational
    routing, intent, score = classify_query_routing(text)
    if routing == "document":
        return None
    if score < 0.65 and intent not in CLEARLY_CONVERSATIONAL:
        return None  # Borderline: send to RAG
    return (intent, score)
PLAIN ENGLISH

classify_conversational_intent decides: is this a greeting/farewell/meta question, or a real document query?

Safety guard: Queries with possessive proper nouns ("Philip's skills", "Gokul's email") are ALWAYS document queries — never misrouted to conversational.

Uses contrastive embedding classification against document-query vs conversational prototypes.

Confidence gate: If the conversational score is below 0.65 and it is not clearly conversational (greeting, farewell, etc.), route to document retrieval. This prevents "Can you show me..." from being misclassified as conversational.

Intents: GREETING, GREETING_RETURN, FAREWELL, THANKS, PRAISE, IDENTITY, CAPABILITY, HOW_IT_WORKS, PRIVACY, LIMITATIONS, USAGE_HELP, SMALL_TALK, CLARIFICATION, DOCUMENT_DISCOVERY.

CODE 
def _search_profile(self, collection, query,
                       query_vector, sub_id, profile_id, top_k):
    qfilter = self._build_filter(sub_id, profile_id)
    result = self.qdrant_client.query_points(
        collection_name=collection,
        query=query_vector, using="content_vector",
        query_filter=qfilter, limit=top_k,
        with_payload=True)
    chunks = [self._point_to_chunk(pt) for pt in result]
    # Keyword fallback when dense returns too few
    high_q = [c for c in chunks if c.score >= 0.5]
    if len(high_q) < 3:
        fallback = self._keyword_fallback(
            collection, query, qfilter, top_k)
        chunks.extend(fallback)
    return chunks
PLAIN ENGLISH

_search_profile — dense vector search for a single profile.

Build a Qdrant filter scoped to the specific subscription + profile (tenant isolation).

Query Qdrant using the "content_vector" named vector field with cosine similarity.

Keyword fallback: If fewer than 3 results score above 0.5 (high quality), trigger a keyword-based scroll search to find matches that vector search missed (e.g., exact names, acronyms).

_BROKEN_HYPHEN_RE — fixes PDF extraction artifacts where "self-employed" becomes "self - employed" (spaces around hyphens).
